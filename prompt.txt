Comportamento esperado dos endpoints:

POST /api/auth/login

Recebe email e senha.
Valida credenciais.
Verifica se usuário está ativo.
Retorna access token, refresh token e dados mínimos do usuário.
Retorna tenants disponíveis para o usuário, sem dados sensíveis.

POST /api/auth/refresh

Recebe refresh token.
Valida assinatura.
Confere hash/registro no banco, se a estratégia usar persistência.
Gera novo access token.
Idealmente rotaciona refresh token.
Rejeita token expirado, revogado ou inválido.

POST /api/auth/logout

Revoga refresh token atual, se persistido.
Retorna sucesso sem vazar detalhes internos.

GET /api/auth/me

Requer access token.
Retorna usuário autenticado e tenants associados.
Não retorna passwordHash.

GET /api/tenants/my

Requer access token.
Retorna apenas tenants em que o usuário autenticado possui vínculo ativo.

Regras de negócio:

Um usuário pode pertencer a múltiplos tenants.
Um tenant pode ter múltiplos usuários.
Um usuário só pode acessar dados de tenants em que possui vínculo ativo.
Usuário inativo não pode fazer login.
Tenant inativo não deve ser tratado como disponível para operação normal.
Senha nunca deve ser salva em texto puro.
Refresh token nunca deve ser salvo em texto puro.
Dados sensíveis como passwordHash nunca devem sair da API.
RBAC deve permitir diferenciar pelo menos admin de plataforma, admin do tenant e membro do tenant.
Toda rota futura de lojista deve ser tenant-scoped.
Nesta tarefa, implementar apenas a base de segurança, não regras comerciais avançadas.
